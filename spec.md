## 2. 계좌 (Account)

> **컨셉**: 계좌는 **노션 페이지 같은 단위**. 사용자가 자유롭게 생성/읽기/쓰기 할 수 있고, 각 페이지에 **속성**(mode, 공유 정책, 카테고리, 목표 연결, 정기입금, 정산일 등)을 부여한다. 공유도 페이지 단위.

### 2.1 속성

모든 계좌는 아래 속성을 가짐 (타입을 별도로 두지 않고, **mode + 옵션 필드** 조합):

- **name**: 사용자 자유 입력 ("맥북 M4 구매", "자취 보증금", "4월 생활비" 등)
- **mode**: `누적형` | `차감형` (2.2)
- **sharing**: `private` | `shared-r` | `shared-rw` (섹션 3)
- **editPolicy**: `author-only` | `any-sharer` — 공유 거래의 **수정** 권한. 삭제는 항상 작성자만.
- **categories**: 계좌별 독립 카테고리 목록
- **goalId** *(optional)*: 특정 목표에 기여하는 계좌인지
- **recurringDeposits** *(optional)*: 정기입금 규칙 0개 이상 (2.3)
- **settlementReminder** *(optional)*: 월말 이월 자동 알림 날짜

### 2.2 mode (핵심)

| mode | 초기잔액 | 기록 의미 | 메인 지표 | 예시 |
|------|----------|-----------|-----------|------|
| **누적형** | 0 | 주로 `+입금`, 필요 시 `-출금`도 가능 | 누적액 / 목표 대비 진행률 | 맥북 구매, 주택자금, 여행 저축 |
| **차감형** | 사용자 설정값 or 정기이체로 충전 | 주로 `-지출` | 남은 잔액 / 사용률 | 이번달 용돈, 이번달 생활비 |

**누적형 예** — 계좌 "맥북 M4"
- 초기 0원 → `+10만` 입금 → 잔액 **10만**
- 또 `+15만` 입금 → 잔액 **25만**
- 목표금액 250만 연결 시 **진행률 10%** 표시

**차감형 예** — 계좌 "4월 용돈"
- 1일에 정기입금 100만 (또는 초기잔액 100만 설정) → 잔액 **100만**
- `-10만` 지출(식사) → 잔액 **90만**
- 메인 UI는 "이번달 남은 90만 / 사용 10만"

> 같은 행위(예: 5만원 지출)여도 누적형/차감형에 따라 **UI 표현**이 달라짐. 내부 자료구조는 모두 `transaction(+/-)` 로 동일.

### 2.3 정기입금 (Income Source)

- 계좌마다 0개 이상의 **정기입금 규칙** 등록 가능
- 규칙 = `{ 주기, 금액, 출처 라벨 }`
- **출처는 사용자 자유 입력**: "월급", "부업 수익", "배당", "이체 from 생활비", "용돈 from 부모님" 등
- 프로토타입: 로그인 시 "마지막 실행 이후 주기가 지났으면 자동 적용"

### 2.4 샘플 시나리오 — 부부(동건/송희)

> 이건 하나의 예시. 실제 사용자는 훨씬 자유롭게 계좌를 만들 수 있음.

**동건의 계좌**

| 이름 | mode | 정기입금 | 목표 연결 | 공유 |
|------|------|----------|-----------|------|
| 아파트 목표(2029) | 누적형 | - | goal_apt_2029 | private |
| 4월 생활비 | 차감형 | 매월 1일 100만 (출처: 월급) | - | shared-rw (송희와) |
| 4월 용돈 | 차감형 | 매월 1일 50만 (출처: 월급) | - | private |
| 주택자금 | 누적형 | 수시 | goal_apt_2029 | shared-r (송희 읽기만) |

**송희의 계좌** (부부 공유 + 본인 개인 용도가 섞임)

| 이름 | mode | 공유 | 비고 |
|------|------|------|------|
| 4월 용돈 | 차감형 | private | |
| 4월 생활비 | 차감형 | shared-rw | 동건의 생활비와 **별개 계좌**, 각자 보유 |
| 주택자금 | 누적형 | shared-r | |
| 자취 보증금 | 누적형 | private | 혼자 쓰는 개인 목적 |
| 맥북 M4 구매 | 누적형 | private | 250만 목표 |

> 생활비는 **각자 1개씩 + shared-rw** 로 공개하는 모델 (공용 1개가 아님). 향후 "공용 계좌"가 필요하면 `ownerId` 대신 `ownerGroupId` 를 허용하도록 확장.

## 3. 공유 정책 (Permission Matrix)

3가지만 우선 지원:

| 정책 | 상대방 조회 | 상대방 입금 | 상대방 지출기록 |
|------|-------------|-------------|-----------------|
| private | X | X | X |
| shared-r | O | X | X |
| shared-rw | O | O | O |

### 3.1 적용 예

- `a_dg_allow` (동건 용돈): private → 송희 화면에서 **리스트/합산/검색 어디에도 노출 안 됨**
- `a_dg_daily` (동건 생활비): shared-rw → 송희가 읽기/입금/지출 전부 가능
- `a_dg_house` (동건 주택자금): shared-r → 송희는 잔액/거래내역만 보임, 지출기록 못 남김
- 반대로 `a_sh_allow` 도 동건에겐 보이지 않음

## 4. 목표 (Goal)

### 4.1 공동 목표

| goal_id | 이름 | 목표 금액 | 목표 시점 | 연결된 계좌 | 소유 |
|---------|------|-----------|-----------|-------------|------|
| goal_apt_2029 | 2029년 아파트 구매 | 현금 2억 | 2029-12 | a_dg_house, a_sh_house, a_dg_goal | 가족그룹 fg_01 |

### 4.2 목표 진행률 계산

```
진행률 = (연결 계좌들의 합산 현재잔액) / 목표금액
```

예: a_dg_house 3,000만 + a_sh_house 2,000만 + a_dg_goal 1,500만 = 6,500만 → **진행률 32.5%**

### 4.3 예상 달성 시점

```
예상달성월 = 현재월 + (목표금액 - 현재잔액) / (월평균 저축액)
```

월평균 저축액 = 최근 3개월 goal-계좌들에 **순증(입금 - 출금)** 평균

## 6. 핵심 시나리오 (계획서의 "소비 이전 의사결정")

### 6.1 S1. 소비 전 영향 조회 (목표 영향 분석)

동건이 250만원짜리 노트북 구매를 고민:

1. 앱에서 **[이 소비 괜찮을까?]** 버튼 → 금액 2,500,000 입력, 지출 계좌 = a_dg_house 지정
2. 앱이 계산:
   - a_dg_house 잔액: 30,000,000 → 27,500,000
   - 가족 목표 연결 자산: 65,000,000 → 62,500,000
   - 월평균 순증 2,400,000 기준 → 아파트 달성 예상 2030-07 → **2030-10 (3개월 지연)**
3. 화면에 표시:
   ```
   노트북 2,500,000 구매 시
   → 2029 아파트 목표 3개월 지연
   ```
4. 동건이 [기록] 또는 [취소] 선택 → 의사결정 UX 완성

### 6.2 S2. 예산 초과 경고 (가계부 모드, 차단 없음 — D6)

동건이 04-20에 식사비로 160,000 결제 시도:
- 현재 식사비 예산 잔여: 150,000
- → **경고 토스트**: "식사비 예산 10,000 초과 (150,000 / 160,000)"
- [그대로 기록] / [재배정하기] / [취소] 세 버튼
- [그대로 기록] 기본값 → 거래는 정상 기록, 카테고리 사용액만 초과 표시(빨강)
- [재배정하기] 선택 시 다른 카테고리에서 10,000 이동 UI 표시

### 6.3 S3. 공유 계좌에서 동시 지출

- 04-15 18:00 송희가 `a_sh_daily`에서 공과금 150,000 기록
- 04-15 18:30 동건이 자신 앱에서 송희의 공과금 지출을 **타임라인**에서 확인
- 동건이 `a_dg_daily`에서 외식 45,000 기록 → 송희 앱에도 즉시(또는 다음 동기화 시) 반영

> 프로토타입: 같은 브라우저/로컬 상태 공유로 시뮬레이션, 실제 다기기 동기화는 DB 붙인 뒤.

### 6.4 S4. 누적 계좌에 수시 입금

- 04-22 동건 보너스 1,000,000 수령 → `a_dg_house`에 수시 입금
- 가족 목표 진행률: 32.5% → 33.0%
- 송희 앱에도 **동건 주택자금 잔액 증가** 가 읽기로 보임
- 송희는 입금 자체를 수정/삭제할 수 없음 (shared-r)

### 6.5 S5. 프라이버시 — 용돈 격리

- 04-25 동건이 `a_dg_allow` 에서 선물 100,000 지출 (카테고리: 여유)
- 송희 앱의 어느 화면에서도 이 거래/계좌/잔액이 노출되지 않음 (검색, 합산, 통계, 타임라인 전부)
- 단, 가족 총자산 계산에는 **포함/제외 여부**를 정책으로 결정 → 기본값: **제외** (개인 용돈은 가족 목표와 무관)

### 6.6 S6. 월말 재배정(Roll with the punches)

- 04-30 송희의 `a_sh_allow` 의 "취미" 예산 100,000 중 40,000만 사용 → 60,000 남음
- 남은 60,000을 다음 달 "여행 저축" 카테고리로 이월할지, 같은 카테고리 유지할지 선택
- 정책: **카테고리 유지 이월 / 다른 카테고리 이동 / 계좌로 반환** 세 옵션

### 7.1 로그인 / 회원가입 / 초대 수락

**기본 로그인 — 로컬에 등록된 사용자 선택**

```
┌──────────────────────────────────────────────┐
│                  🥧 one-pie                   │
│        목표 기반 재무 의사결정 도구            │
│                                              │
│   ┌────────────────┐  ┌────────────────┐     │
│   │   👨  동건     │  │   👩  송희     │     │

│                                              │
│         [ + 새 계정 만들기 ]                  │
└──────────────────────────────────────────────┘
```
- 프로토타입: 계정/비번 없이 프로필 카드 클릭 시 해당 사용자 세션 진입
- localStorage `currentUserId` 저장
- 로컬에 등록된 모든 사용자 = 동일 브라우저에서 가입/초대 수락한 사용자 + 시드 데이터

**회원가입 (SignUp) — [+ 새 계정 만들기] 클릭**

```
┌──── 새 계정 ────────[×]─┐
│                          │
│  이름 [ 홍길동       ]   │

```

- 가입 시 해당 사용자를 **소유자로 하는 새 `FamilyGroup` 자동 생성** (혼자인 상태)
- 저장 후 자동 로그인 → 대시보드 진입


### 7.9 설정 — 사용자 전체

카카오톡 스타일 **메뉴 리스트**. 최상위는 프로필 카드 + 항목 리스트. 각 항목은 탭하면 하위 페이지로 진입.

```
┌─ ⚙ 설정 ──────────────────────────────┐
│                                        │
│  ┌──────────────────────────────┐     │
│  │ 👨  동건                       │     │
│  │     가족 그룹: fg_01           │     │
│  └──────────────────────────────┘     │
│                                        │
│  ┌──────────────────────────────┐     │
│  │ 💳  내 계좌         4개    ›  │     │
│  ├──────────────────────────────┤     │
│  │ 🎯  목표            3개    ›  │     │
│  ├──────────────────────────────┤     │
│  │ 🔁  반복 거래       2개 규칙 › │     │
│  ├──────────────────────────────┤     │
│  │ 👨‍👩‍👧 가족   동건·송희 · 2명 ›  │     │
│  ├──────────────────────────────┤     │
│  │ 🛠  개발자                 ›  │     │
│  └──────────────────────────────┘     │
└────────────────────────────────────────┘
```

- 라우트
  - `/settings` : 메뉴 리스트 (프로필 카드 + 항목 리스트)
  - `/settings/accounts` : 내 계좌 관리 (드래그 정렬, 열기/속성, + 새 계좌)
  - `/settings/goals` : 목표 관리 (열기/편집, + 새 목표)
  - `/settings/recurring` : 반복 거래 규칙 (편집, + 새 규칙, 🔄 지금 적용)
  - `/settings/family` : 가족 그룹 정보, 구성원, + 가족 초대 (QR)
  - `/settings/developer` : 샘플 데이터 재로드, 전체 초기화
- 하위 페이지는 공통으로 상단에 `← 설정` 뒤로가기 버튼 제공

**가족 초대 모달 (QR)** — [+ 가족 구성원 초대] 클릭 시:

```
┌──── 👥 가족 초대 ─────────[×]─┐
│                                │
│   ┌────────────────────┐       │
│   │   ▇▇ ▇▇ ▇▇ ▇▇ ▇▇   │       │
│   │   QR 코드 이미지    │       │
│   │   ▇▇ ▇▇ ▇▇ ▇▇ ▇▇   │       │
│   └────────────────────┘       │
│                                │
│  가족 그룹: 동건·송희           │
│  초대한 사람: 👨 동건            │
│                                │
│  초대 링크                      │
│  [ https://.../?invite=... ]  │
│  [ 링크 복사 ]                  │
│                                │
│  상대방이 QR을 스캔하거나        │
│  링크를 열면 가족으로 합류       │
└────────────────────────────────┘
```
- QR 이미지는 `qrcode` 라이브러리로 canvas 렌더
- 초대 토큰 = base64url(JSON({ v:1, familyGroupId, familyGroupName, inviter: {id, name, emoji?} }))
- 링크 형식: `${location.origin}${location.pathname}?invite=<token>`
- 스캔/링크 오픈한 기기(또는 같은 브라우저의 다른 세션)에서 7.1의 "초대 수락" 플로우 진입

### 7.11 카테고리 택소노미

거래의 `category` 필드에 담기는 값들은 **전역 카테고리 트리**에서 고른 서브 label. 계좌별로 카테고리를 매번 만들지 않아도 되도록 사용자 단위로 공유.

- 구조: `MainCategory[]` — 각 메인은 `kind: 'expense' | 'income'`, `emoji`, `label`, `subs: SubCategory[]`
- 기본값: `utils/categoryDefaults.ts`의 `DEFAULT_CATEGORY_TAXONOMY` (지출 10 메인 · 수입 7 메인). 신규 가입/초기화/기존 persist 상태에 없을 때 자동 주입.
- 편집: `/settings/categories` — 카카오톡 스타일 2-pane (지출/수입 탭 → 좌측 메인 리스트 → 우측 서브 리스트). 메인은 emoji+label 편집·삭제, 서브는 label 편집·삭제. `↺ 기본값` 버튼으로 공장 초기화.
- 거래 모달의 카테고리 필드: 현재 선택된 서브를 칩으로 표시하고, 탭하면 bottom-sheet 피커 오픈 → 지출/수입 탭 → 메인 칩 탭 → 서브 칩 탭 → 자동 반영·닫힘. 피커 상단 `⚙` 아이콘으로 `/settings/categories` 이동.
- 기존 `Account.categories` + `budgetAllocations`는 계좌별 예산 배정용으로 유지 (별개 개념). 거래의 카테고리 서브 label이 예산 배정 키와 일치하면 잔여 예산 힌트가 표시됨.

```ts
  settlementReminder?: {            // 월말 이월 자동 알림 (D13-a)

  }
}

type Budget = {                     // 주로 차감형 계좌에서 사용 (envelope)

  month: string                     // '2026-04'
  categories: { name: string; allocated: number; spent: number }[]
}

type Transaction = {
  id: string
  accountId: string
  authorId: string                  // 실제 기록한 사용자 (공유 계좌에서 누가 썼는지)
  date: string
  amount: number                    // +입금 / -지출
  category?: string                 // 차감형 예산 카테고리
  source?: string                   // 입금 시 출처 라벨 (정기입금 규칙에서 상속 또는 수기 입력)
  memo?: string
}

type Goal = {
  id: string
  ownerType: 'user' | 'family'

  targetAmount: number
  targetDate: string
  linkedAccountIds: string[]
```

## 10. 결정된 사항

- **D1.** 계좌는 목적 자유 (부부/개인/임의 목적 모두 허용). 부부 케이스는 샘플 시나리오.
- **D2.** 계좌마다 **mode(누적형/차감형)** 를 사용자가 지정. UI 표현만 달라지고 데이터 모델은 통일.
- **D3.** 정기입금의 **출처 라벨(월급/배당/…)** 도 사용자가 자유 입력.
- **D4.** 용돈(private 계좌)은 **가족 총자산/목표 계산에서 제외**가 기본값 (goalId 연결 안 된 private 계좌는 가족 합산에서 빠짐).
- **D5.** 생활비는 부부 각자 1개씩 보유 + shared-rw 로 상호 공개. 향후 "공용 계좌"가 필요하면 `ownerGroupId` 로 확장.
- **D6.** 차감형 예산 **초과 시 경고만** 표시하고 소비 자체는 기록 허용. 가계부 성격이므로 차단하지 않음. (재배정은 사용자가 원할 때 수동)
- **D7.** 공유 계좌의 거래 **수정/삭제 권한은 계좌 생성 시점에 선택**. 옵션: `author-only` (작성자만) / `any-sharer` (shared-rw 권한자 누구나). 계좌 설정에서 이후 변경 가능.
- **D8.** 차감형 계좌의 **월말 이월은 사용자 선택**. 월말(또는 다음달 진입 시) 카테고리별로 3옵션 제시: (a) 같은 카테고리로 이월 / (b) 다른 카테고리로 이동 / (c) 계좌 잔액으로 반환.
- **D9.** **누적형 계좌도 카테고리를 가짐**. 차감형=지출 카테고리 중심, 누적형=입금 카테고리 중심(예: "월급 적립", "중고판매", "부모님 지원"). 데이터상 Budget/Transaction은 동일, UI 표기만 다름.
- **D10.** `editPolicy`는 **수정에만** 적용. **삭제는 항상 `author-only` 로 고정** (공유자 실수로 타인의 거래 삭제 방지).
- **D11.** 누적형 카테고리는 **계좌별 독립**. 계좌 간 카테고리 풀을 공유하지 않음.
- **D12. 계좌 = "노션 페이지" 모델.**
  - 사용자는 계좌(=페이지)를 자유롭게 생성
  - 각 페이지에 **속성** 을 부여: mode(누적/차감), sharing(private/shared-r/shared-rw), editPolicy, goal 연결, 카테고리 목록, 정기입금 규칙, 월말 정산일 등
  - 공유 여부는 **페이지 단위**. 독립 계좌와 공유 계좌가 한 사용자 내에 자유롭게 섞임
  - UI도 페이지 리스트 + 속성 편집 패널 구조로 설계
- **D13.** 월말 이월 트리거는 **두 가지 병행**:
  - (a) 사용자가 달력에서 지정한 날짜(예: 매월 말일, 매월 25일)에 **자동 알림** → 알림에서 바로 정산 화면 진입
  - (b) 사용자가 계좌 화면에서 [정산하기] 버튼을 누르면 즉시 정산 화면 진입
  - 둘 다 같은 정산 UI로 연결 (카테고리별 유지/이동/반환 선택 — D8)
- **D18. 계좌 "월별 관리" 모델**
  - 계좌는 월 단위로 복제되지 않고 **영속 객체 1개**. 대신 모든 월 의존 데이터는 `month: 'YYYY-MM'` 키로 관리:
    - `budgetAllocations[month]` — 월별 카테고리 배정
    - `monthlyBudget[month]` — 월별 총 예산
    - 거래(Transaction)의 `date`
  - AccountDetail 상단에 `MonthNavigator` (‹ 2026년 4월 › · 오늘로). 월을 바꾸면:
    - 차감형 요약(초기배정/추경/지출/남음) 모두 해당 월로 재계산
    - 카테고리 행의 배정/사용 모두 해당 월 기준
    - 거래 목록은 해당 월 거래만 표시 (빈 달이면 "다른 달에는 있음" 힌트)
    - PropertyPanel 열면 해당 월의 예산/카테고리 배정을 편집
    - SettlementModal도 해당 월 기준
  - **편의 기능**: 빈 달에 PropertyPanel 열었을 때 "📋 이전달 복사" 버튼 → 바로 이전 달의 `monthlyBudget`/`budgetAllocations`를 복사 (저장 전 로컬 state에 적용, 저장 누르면 반영).
  - 누적형 계좌는 잔액이 시간에 걸쳐 누적되므로 **잔액 자체는 월 무관**. 단 거래 목록만 월별 필터. (카테고리별 입금 집계는 추후 월별 뷰로 확장 가능)
- **D17. 계좌 종류(type)**
  - 필드: `Account.type?: AccountType` (선택적, 기본값 `'계좌'` — 기존 데이터 호환).
  - 값: `'현금' | '계좌' | '체크카드' | '현금카드'`.
  - **mode와 직교**: type(결제/보유 수단) vs mode(누적형/차감형)는 독립. 예) "현금 + 차감형" = 지갑 속 월 용돈 / "계좌 + 누적형" = 저축 / "체크카드 + 차감형" = 카드 생활비.
  - UI: PropertyPanel 최상단에 종류 라디오, 각 카드(Dashboard/AccountDetail/Settings)에 이모지+라벨 뱃지 표시.
  - 메타 테이블(`ACCOUNT_TYPE_META`)에 이모지/라벨/힌트 중앙 관리 → 향후 타입 추가/재조정 쉬움.
- **D16. 카테고리 UI/데이터 모델**
  - **UI 분리**: 카테고리 **선택**(거래 기록 / 속성 표시)과 **관리**(추가/수정/삭제)를 명확히 분리.
    - 선택: 칩(chip) 컴포넌트 `CategoryChips` — 단순 탭. 추가/편집 UI 없음.
    - 관리: 별도 모달 `CategoryManagerModal` (⚙ "설정" 버튼으로 진입). 행 단위로 ✏ 편집 → rename/delete/cancel. 하단에 추가 입력.
    - 거래 기록 모달에는 ⚙ 버튼 추가. 자유 입력으로 즉석 추가하는 UX는 제거(항상 관리 모달 경유).
  - **즉시 저장**: 카테고리 관리 액션(add/rename/delete)은 즉시 store에 반영. rename 시 `budgetAllocations` 키와 관련 `transactions.category` 자동 동기화. delete 시 해당 카테고리의 배정 제거, 관련 거래의 category는 `undefined`로 비움.
  - **현재 데이터 모델**: `account.categories: string[]` — 계좌별 독립.
  - **확장 포인트(향후)**:
    - `Category = { id, name, emoji?, kind?: 'expense' | 'income' }` 분리
    - 공통 풀(user 또는 familyGroup 범위) + 계좌별 opt-in (`account.enabledCategoryIds`)
    - 이 전환 시에도 UI(CategoryChips/CategoryManagerModal)는 재사용 — store 액션(`addCategory` 등)만 풀 기반으로 교체하면 됨.
    - TransactionModal/PropertyPanel은 `account.categories` 하나만 읽으므로, 향후 selector로 감싸 대체 가능.
- **D15. 차감형 계좌 예산 모델 — "초기배정 + 추경"**
  - **초기배정(initialAllocated)**: 다음 중 큰 값.
    - `Account.monthlyBudget[month]` — 사용자가 PropertyPanel에서 직접 입력한 "이번달 총 예산" (덩어리 예산, 카테고리 없이 총액만 잡을 때 유용).
    - `sum(budgetAllocations[month])` — 카테고리별 배정 합계 (계획분).
  - 둘 다 있으면 **max**(총 예산 vs 카테고리 합). 카테고리만 있고 총액은 빈 경우 → 카테고리 합. 총액만 있고 카테고리 없음 → 총액. 둘 다 비어있으면 0.
  - **추경(supplemented)**: 이번 달 거래 중 **`isSupplement === true` 로 체크된 입금만** 합산. 명시적으로 "예산에 넣는다"고 표시한 입금.
  - **일반 입금(regularInflow)**: `isSupplement` 플래그 없는 양수 거래(이자/월급/부수입 기록용). **예산에 반영되지 않고** 기록만 남음.
  - **총 예산(allocated)** = initialAllocated + supplemented.
  - **남음(remaining)** = allocated − spent.
  - **UI 규칙 — 두 개의 진행 바**
    - 메인 바: 초기배정 대비 사용률. 초과 시 100% + 빨강.
    - 보조 바(추경이 있을 때만): 추경 대비 초과 지출(`max(0, spent - initialAllocated)`) 사용률. 추경도 초과 시 빨강.
    - 두 바의 색상은 초기(초록/노랑/빨강) / 추경(파랑, 초과 시 빨강)로 시각 구분.
  - **왜**: "계좌에 돈 넣었는데 예산 반영이 안된다" 혼란 해결 + 예산에 합산될 돈과 기록용 수입을 명시적으로 구분. 사용자가 "이 돈은 쓰려고 추가한 건가 vs 기록만 남기는 건가"를 거래 입력 시 결정하도록 강제.
  - **거래 입력 UX**: 차감형 + 입금 선택 시 "💰 추경 — 이 입금을 이번달 예산에 합산" 체크박스 노출. 기본 off. 거래 목록에도 💰 추경 뱃지 표시.
  - 누적형 계좌는 이 개념 미적용 (누적형은 모든 입금이 잔액에 누적되므로 구분 불필요).
  - 알림 계산(D14)은 `remaining`/`allocated`(=initial+supplemented) 기준 — 추경이 들어오면 자동으로 알람 완화.
- **D14. 차감형 계좌 "잔액 임계 알림"**
  - 속성: `lowBalanceAlert?: { amount?: number; percent?: number; notify?: 'owner' | 'all' }`. 임계치는 둘 다 선택적(미설정 가능), 둘 다 설정 시 **OR** 조건(둘 중 하나라도 도달하면 알림).
  - **알림 대상 (`notify`)**: `owner` = 계좌 소유자만 (기본), `all` = 해당 계좌를 볼 수 있는 공유자 모두. private 계좌는 실질적으로 owner만.
  - **주 전달: 대시보드 상단 토스트** — 앱이 띄워진 모든 화면에서 상단에 표시, `X` 버튼으로 **이번 달 동안 숨김**(다음 달 진입 시 재등장). 구현상 `localStorage`에 `one-pie-lowbal-dismissed-{YYYY-MM}` 키로 숨김 ID 목록 저장.
  - **보조 전달: 거래 모달 인라인 힌트** — 지출 기록 시 "이 지출 후 남음 / 임계치 도달" 안내(차단 없음). 알림 대상 규칙 동일(대상 아니면 힌트도 안 보임).
  - 조건: 차감형 + 배정 > 0 + 임계치 설정됨. 잔액이 음수면 🚨(danger) 스타일, 임계 이하 양수면 ⚠(warn) 스타일.
  - 누적형은 "목표(Goal)" 체계로 대체하므로 대상 외.

- **D19. 회원가입 / 초대 (프로토타입)**
  - 회원가입 = **이름 + 이모지만** 입력. 비번/이메일/소셜 로그인 없음. DB 붙이기 전 최소 UX.
  - 가입 시 해당 사용자를 소유자로 하는 **새 `FamilyGroup` 자동 생성**. 이후 초대/가입 시 이 그룹에 합류.
  - 초대 방식: 가족 그룹 소유자가 **QR/링크** 발급. 토큰은 `{ v, familyGroupId, familyGroupName, inviter: {id, name, emoji} }` 의 base64url JSON. URL 쿼리 `?invite=<token>`.
  - 수락: 동일 브라우저(localStorage)에서 링크를 열면 Login 화면이 초대 수락 UI로 전환. 이름 입력 → 신규 User 생성 + familyGroupId 합류. inviter 가 로컬에 없으면 토큰 정보로 **stub User** 를 만들어 그룹에 추가(공유 계좌 ownership 표시 유지).
  - 크로스 디바이스는 프로토타입 범위 밖 (같은 localStorage 내부에서 여러 User 를 두어 공유 시나리오를 시뮬레이션). DB 붙이는 단계에서 동일 토큰 구조를 서버 초대로 교체.
  - 시드 데이터의 `동건 / 송희 / fg_01`은 유지 — 신규 가입자는 이와 독립된 계정/그룹으로 시작.

- **D20. 통계 페이지 월별 필터**
  - Stats 페이지 상단에도 `MonthNavigator` 추가 (Dashboard / AccountDetail 와 동일 컴포넌트).
  - **선택 월 기준으로 재계산되는 섹션**: 월별 요약 카드(수입/지출/순이익/저축률), 카테고리별 지출 도넛, 결제 수단별 지출 막대.

- **D21. 반복 거래 규칙 (Recurring Rule)**
  - 기존 `Account.recurringDeposits`(계좌 속성 기반 정기입금)와 별개로, **사용자 단위 반복 규칙**을 추가. 규칙은 특정 계좌에 지출/입금을 주기적으로 자동 생성함.
  - 설정 페이지에 "반복 거래" 섹션 → 규칙 리스트 + 추가/편집/삭제 모달(`RecurringRuleModal`).
  - **주기 옵션**: `daily | weekdays | weekends | weekly | biweekly | every4weeks | monthly | every2months | every3months | every4months | every5months | every6months | yearly`.
  - **필드**: `ownerId`, `accountId`, `kind('expense'|'deposit')`, `amount`(양수), `interval`, `startDate`, `endDate?`, `category?`, `source?`, `memo?`, `isSupplement?`, `lastRunDate?`, `enabled`.
  - **자동 생성(materialize)**: 앱 진입/로그인 시 `materializeRecurringRules()` 호출. 각 규칙별로 `(lastRunDate || 시작일)`부터 `min(오늘, endDate)`까지 주기 발생일을 계산해 `Transaction`을 생성(`autoGenerated: true`, `sourceRuleId: rule.id`). 생성 후 `lastRunDate` 갱신.
  - **중복 방지**: `lastRunDate` 이후 발생일만 생성하므로 같은 날 재실행해도 중복 생성 없음.
  - **삭제 시**: "자동 생성된 거래도 함께 삭제" 여부를 사용자가 선택. 규칙만 삭제해도 지난 거래는 유지 가능.
  - **종료일 없음** = 무기한. 규칙을 비활성화(`enabled=false`)하면 자동 생성 중단(기록은 유지).
  - **왜**: "매달 월급 자동 기록", "매주 구독료", "매년 보험료" 같은 반복 입력 자동화. 사용자가 매번 거래를 수동 기록하지 않도록 편의성 향상.
